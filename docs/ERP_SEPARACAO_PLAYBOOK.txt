ERP SEPARADO DO PONTO CERTO - PLAYBOOK OPERACIONAL
Data: 2026-04-11

Nome provisorio do novo produto
- ContaFlux ERP

Objetivo
- Criar um ERP separado do Ponto Certo.
- Subir em outro dominio.
- Usar outro projeto Firebase.
- Reaproveitar apenas a base util: contador, financeiro, fiscal, clientes e servicos.
- Incluir fluxo de admissao digital de funcionario conduzido pelo contador.
- Evitar carregar modulos de ponto, funcionarios, jornada e fluxos especificos do Ponto Certo.

Decisao recomendada
- Novo repositorio.
- Novo projeto Firebase.
- Nova identidade visual.
- Nova navegacao.
- Mesmo padrao tecnico que ja funciona hoje.

Fluxo ideal de emissao de nota
1. Empresa cadastra servico uma vez.
2. Empresa cadastra ou seleciona cliente.
3. Em /notas escolhe o servico.
4. Sistema preenche descricao fiscal, codigo de servico, CNAE, aliquota e observacoes padrao.
5. Usuario informa valor e confirma.
6. Nota fica salva com historico e status.
7. Contador acompanha em /contador.

Fluxo ideal de admissao digital do funcionario
1. Empresa ou contador abre /contador/admissoes.
2. Informa nome, e-mail, cargo, data de admissao e checklist documental.
3. Sistema gera um convite com token unico e validade.
4. Sistema envia e-mail ao funcionario com orientacoes e link seguro.
5. Funcionario acessa /admissao/{token}.
6. Funcionario preenche os dados pessoais e profissionais solicitados.
7. Funcionario envia os documentos obrigatorios.
8. Sistema salva tudo separado por empresa e por pedido de admissao.
9. Contador revisa pendencias sem depender de WhatsApp ou visita presencial.
10. Quando tudo estiver correto, a admissao vai para status pronta para fichar.

Objetivo da admissao digital
- eliminar conversas perdidas em WhatsApp
- evitar documentos soltos e mistura de informacoes
- padronizar o onboarding trabalhista
- deixar cada empresa isolada no proprio ambiente
- permitir rastreabilidade completa do envio e da revisao

Passo a passo recomendado

Fase 1 - Criar novo repositorio
1. Criar uma nova pasta fora deste projeto.
2. Inicializar novo app Flutter web-first.
3. Copiar somente os modulos que entram no MVP.
4. Renomear pacote, titulo web e identidade visual.

Comandos base
flutter create contaflux_erp
cd contaflux_erp
git init

Fase 2 - Criar o novo projeto Firebase
Observacao importante
- A maior parte da configuracao pode ser feita via CLI.
- Custom domain normalmente exige etapa final no Firebase Hosting e configuracao DNS no provedor do dominio.
- Login no Google, criacao do projeto no GCP em alguns ambientes e ajustes de Auth provider podem abrir navegador ou exigir console.

Pre-requisitos de CLI
- Node.js instalado
- Firebase CLI instalada
- FlutterFire CLI instalada

Comandos de preparo
npm install -g firebase-tools
dart pub global activate flutterfire_cli
firebase login

Criar o projeto Firebase
Opcao A - criar no console e depois conectar via CLI
1. Criar o projeto no console com nome ContaFlux ERP.
2. Definir um project id, exemplo: contaflux-erp-prod.
3. Voltar para o terminal e conectar.

Comandos
firebase use --add

Opcao B - tentar via CLI/GCloud
Observacao
- Isso depende de ter gcloud instalado e autenticado.
- Se nao estiver, a opcao A e mais rapida e segura.

Exemplo
gcloud projects create contaflux-erp-prod --name="ContaFlux ERP"

Habilitar servicos principais
firebase projects:list
firebase use contaflux-erp-prod

Inicializar o Firebase no projeto novo
firebase init hosting
firebase init firestore
firebase init storage
firebase init functions

Respostas recomendadas
- Use existing project: contaflux-erp-prod
- Hosting public dir: build/web
- Single-page app: yes
- Automatic builds with GitHub: no, por enquanto
- Functions: JavaScript ou TypeScript, conforme preferencia
- Runtime: nodejs22

Fase 3 - Vincular Flutter com o novo Firebase
No novo projeto Flutter:
flutterfire configure --project=contaflux-erp-prod

Se quiser web e android no primeiro corte:
flutterfire configure --project=contaflux-erp-prod --platforms=web,android

Regra obrigatoria de isolamento
- Toda admissao deve carregar companyId.
- Toda consulta, upload e leitura devem ser filtrados por companyId.
- O token publico de admissao deve apontar para apenas uma empresa e um pedido especifico.
- Um funcionario convidado nunca pode listar dados de outros funcionarios.
- O contador acessa apenas empresas vinculadas ao proprio usuario.
- Nao pode existir compartilhamento de documentos entre empresas.

Fase 8 - Admissao digital pelo contador
Objetivo
- Centralizar a coleta de dados e documentos admissionais no ERP.
- Reduzir retrabalho, perda de contexto e troca de mensagens fora do sistema.

Fluxo
1. Contador ou empresa cria uma admissao em /contador/admissoes.
2. Sistema gera token seguro com expiracao.
3. Cloud Function envia e-mail com orientacoes e link exclusivo.
4. Funcionario abre o link e preenche os dados.
5. Funcionario sobe os documentos solicitados.
6. Sistema grava arquivos em pasta segregada por empresa e pedido.
7. Contador revisa, aprova ou devolve com pendencias.
8. Empresa acompanha o status sem acessar dados de outras empresas.

Rotas envolvidas
- /contador/admissoes
- /contador/admissoes/nova
- /contador/admissoes/{id}
- /admissao/{token}

Cloud Functions recomendadas
- createHiringRequest
- sendHiringInviteEmail
- validateHiringToken
- submitHiringForm
- reviewHiringRequest

Storage path recomendado
- companies/{companyId}/hiring/{requestId}/{documentType}/{fileName}

Beneficio operacional
- elimina mistura de documentos
- elimina dependencia de conversa espalhada
- reduz ida presencial ao escritorio so para entrega de papel
- cria trilha auditavel por empresa
- deixa o contador trabalhar com fila organizada de admissoes

Ordem pratica de execucao
1. Definir nome final do ERP.
2. Criar repositorio novo.
3. Criar projeto Firebase novo.
4. Rodar flutterfire configure.
5. Copiar core minimo e modulos do MVP.
6. Ajustar rotas e RBAC.
7. Criar collections e rules.
8. Implementar fluxo de admissao digital.
9. Subir web em dominio temporario do Firebase.
10. Validar fluxo de login, clientes, servicos, financeiro, notas e admissoes.
11. Conectar dominio proprio.

Modelo comercial inicial
- Estrategia recomendada: plano unico.
- Mesmos recursos para todos.
- Diferenca de preco apenas pelo porte do cliente.

Preco sugerido para lancamento
- MEI: R$ 59 por mes
- Empresa: R$ 99 por mes

Regra comercial
- O plano e o mesmo.
- Nao separar por modulo.
- Nao separar por quantidade de recursos no inicio.
- Diferenciar apenas entre MEI e Empresa por capacidade de pagamento e faturamento.

Justificativa da estrategia
- simplifica a venda
- simplifica o site
- simplifica o suporte
- facilita indicacao por contadores
- reduz friccao comercial
- aumenta chance de ganhar base mais rapido

Posicionamento comercial sugerido
- Um unico plano para organizar financeiro, servicos, notas e contador em um so lugar.
- R$ 59 por mes para MEI.
- R$ 99 por mes para empresa.
- Mesmos recursos, com valor ajustado ao porte do negocio.

Texto curto para usar no site
- Plano unico, simples e direto.
- MEI paga R$ 59 por mes.
- Empresa paga R$ 99 por mes.
- Todos usam os mesmos recursos: financeiro, servicos, emissao de notas, painel do contador e admissao digital.

Estrutura de parceria com contadores
- O contador parceiro indica empresas para entrar no sistema.
- O cliente entra no plano unico conforme porte.
- O contador recebe beneficio comercial ou operacional.

Modelos simples de parceria
- acesso gratuito ao painel do contador
- implantacao assistida para a carteira indicada
- desconto por volume de empresas ativas
- comissao recorrente por cliente ativo

Recomendacao para o inicio
- nao criar muitos formatos de parceria agora
- escolher uma regra simples e facil de operar
- exemplo: contador parceiro com painel gratuito e beneficio por volume de clientes ativos

Observacao de precificacao
- Esses valores fazem mais sentido para entrada de mercado e ganho de base.
- Quando o produto estiver validado, pode existir reajuste para novas entradas sem mexer na base inicial.
